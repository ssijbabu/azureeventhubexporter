# Azure Event Hub Exporter — Contribution Plan

This file is local only (excluded via .git/info/exclude). Do not commit it.

The current PR (#49203) and issue (#49204) on open-telemetry/opentelemetry-collector-contrib
were opened prematurely. The project's `docs/new-components.md` requires a defined process
before a component can be accepted. Follow the steps below in order.

---

## Phase 1 — Close the premature PR and issue

1. Go to https://github.com/open-telemetry/opentelemetry-collector-contrib/pull/49203
   - Click **Close pull request** (do not delete the branch yet — you still need the code).

2. Go to https://github.com/open-telemetry/opentelemetry-collector-contrib/issues/49204
   - Click **Close issue**.
   - Optional: leave a comment explaining you are following the proper donation process.
     (Write the comment yourself — do not use AI-generated text in issue/PR discussions
     on this repo per AGENTS.md.)

---

## Phase 2 — Host the component outside the repo first

The project requires external hosting before a donation PR is accepted.

1. Create a new public GitHub repository, e.g. `ssijbabu/azureeventhubexporter`.

2. Copy the current exporter code into it:
   ```
   exporter/azureeventhubexporter/   →   root of the new repo
   ```
   Strip out the `replace` directives from `go.mod` that point to the monorepo,
   and change the module path to match your own repo.

3. Write a short README in the new repo explaining:
   - What it does
   - How to use it
   - Authentication options
   - Partition strategies

4. Publish at least one release tag (e.g. `v0.1.0`) so it is importable as a Go module:
   ```bash
   git tag v0.1.0
   git push origin v0.1.0
   ```

5. Add the component to the OpenTelemetry registry:
   https://opentelemetry.io/ecosystem/registry/adding/
   This increases visibility and shows community traction.

6. Gather usage and feedback. The more users and stars, the stronger the donation case.
   Aim for at least a few weeks of external availability.

---

## Phase 3 — Find a sponsor

The project requires a sponsor: an approver or maintainer from a different company
who agrees to review the code and take co-responsibility.

1. Join CNCF Slack: https://slack.cncf.io  
   Channel: `#otel-collector-dev`

2. Post a message announcing your intention to donate the component. Example:
   > "I've built an Azure Event Hub exporter for the OTel Collector
   > (https://github.com/ssijbabu/azureeventhubexporter). It exports logs, metrics,
   > and traces to Azure Event Hubs via OTLP JSON. Looking for a sponsor to help donate
   > it to opentelemetry-collector-contrib. Is anyone interested?"

3. Attend a Collector SIG meeting to raise visibility.
   Meeting schedule: https://github.com/open-telemetry/community#special-interest-groups

4. The existing receiver codeowners (@atoulme, @cparkins, @dyl10s) are natural candidates
   since they already own the sibling component. Reach out to them directly but note the
   sponsor must be from a different company than you.

5. Do not proceed to Phase 4 until you have at least one confirmed sponsor.

---

## Phase 4 — Open the official new-component issue

Use the official template, not a free-form issue.

1. Go to:
   https://github.com/open-telemetry/opentelemetry-collector-contrib/issues/new?assignees=&labels=Sponsor+Needed%2Cneeds+triage&projects=&template=new_component.yaml&title=Component+donation%3A+

2. Fill in all required fields:
   - **Sponsor**: GitHub handle of the approver/maintainer who agreed in Phase 3.
   - **Codeowners**: At least 3 GitHub handles. At least one must be an approver/maintainer.
     They should ideally not all work for the same company.
     Candidates: your sponsor + @atoulme + @cparkins or @dyl10s (confirm their availability).
   - **Competing components**: List `receiver/azureeventhubreceiver` (sibling component),
     and note there is no existing exporter. Mention Kafka exporter as an analogous
     component for a different broker.
   - **Component description**: Explain the use cases (fan-out pipelines, Azure integrations,
     cross-region buffering).
   - **Configuration options**: Reference the config fields in `config.go`.

3. Write the issue yourself — do not use AI-generated text per AGENTS.md.

4. Wait for maintainer approval before opening a code PR.

## Phase 5 — Prepare the code for the donation PR

Once the issue is approved, bring the code up to the repo's standards.
Run all of the following from the root of the repo (not the component directory).

### 5.1 — Set codeowners in metadata.yaml

Edit `exporter/azureeventhubexporter/metadata.yaml`:
```yaml
  codeowners:
    active: [<sponsor-handle>, <codeowner-2>, <codeowner-3>]
```

### 5.2 — Fix README.md header

The README must start with the autogenerated status block. Replace the current title line:
```markdown
<!-- status autogenerated section -->
# Azure Event Hub Exporter
| Status        |           |
| ------------- |-----------|
| Stability     | [development]: metrics, logs, traces |
| Distributions | [] |
| Issues        | ... |
| [Code Owners] | @sponsor @codeowner2 @codeowner3 |
<!-- end autogenerated section -->
```
Running `make generate` in step 5.4 will regenerate this automatically.

### 5.3 — Run crosslink (wires intra-repo replace directives)

```bash
make crosslink
```

This adds/updates `replace` directives in the `go.mod` files of every package
in the monorepo that depends on the new exporter, so it builds correctly inside contrib.

### 5.4 — Run the metadata generator

```bash
cd exporter/azureeventhubexporter
make generate
```

This runs `mdatagen` against `metadata.yaml` and regenerates:
- `internal/metadata/generated_status.go` (replaces the hand-written version)
- README.md status header

After running, delete the hand-written `internal/metadata/generated_status.go`
and replace it with the generated one.

### 5.5 — Regenerate CODEOWNERS

```bash
make gencodeowners
```

This adds the new exporter to `.github/CODEOWNERS`.

### 5.6 — Add to builder-config.yaml (required for alpha, but do it now)

Edit `cmd/otelcontribcol/builder-config.yaml` and add under `exporters:`:
```yaml
  - gomod: github.com/open-telemetry/opentelemetry-collector-contrib/exporter/azureeventhubexporter v0.154.0
```
Match the version number to the other entries already in that file.

### 5.7 — Run all required pre-PR validation commands

Run each of the following from the repo root. Fix any errors before continuing.

```bash
make checkdoc
make checkmetadata
make checkapi
make goporto
make tidylist
make gotidy
make multimod-verify
make generate-gh-issue-templates
make gengithub
make genlabels
make gencodecov
make addlicense
make genotelcontribcol
make genoteltestbedcol
```

Note: some of these may take several minutes. Run `make lint` as well:
```bash
cd exporter/azureeventhubexporter && make lint
```

### 5.8 — Run tests from the repo root

```bash
make gotest
```

Or targeted:
```bash
cd exporter/azureeventhubexporter && make test
```

## Phase 6 — Open the donation PR

Structure the PR as two separate PRs per the docs:

### PR 1 — Structure (README, config, factory)

Include:
- `exporter/azureeventhubexporter/` with all files
- `versions.yaml`
- `.chloggen/` entry
- `.chloggen/config.yaml` (exporter/azure_event_hub added)
- `.github/CODEOWNERS` (generated)
- `cmd/otelcontribcol/builder-config.yaml`
- All generated files from Phase 5

PR title format: `[exporter/azureeventhubexporter] Add Azure Event Hub exporter`

PR description must include:
- `Closes #<issue-number-from-phase-4>`
- Summary of what the component does
- Test plan checklist
- `Assisted-by: Claude Sonnet 4.6` trailer in the commit message

Tag the sponsor and proposed codeowners in the PR description so they see it.

### PR 2 — Concrete implementation

If the implementation is large enough to warrant a separate PR, split it here.
In our case the implementation is already included in PR 1 code, so this may not be needed.

### PR 3 — Alpha promotion (later)

After the component has been in development for a while and is stable:
- Update `metadata.yaml`: change `development` → `alpha`
- Add `contrib` to distributions list
- Run `make generate` and `make genotelcontribcol`
- Open a PR to `opentelemetry-collector-releases` to include it in the otelcol-contrib binary

## Phase 7 — Post-merge

1. Add the component to the OpenTelemetry registry once it is in the contrib binary:
   https://github.com/open-telemetry/opentelemetry.io#adding-a-project-to-the-opentelemetry-registry

2. Maintain the component: respond to issues, review PRs, keep dependencies updated.
   Failing to maintain it may result in it being marked unmaintained and removed.
